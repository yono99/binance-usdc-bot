# Plan Item 4 — Open Orders live dari Binance + perbaikan bug limit-order resting

## Akar masalah
1. `exchange.py` tak punya `open_orders()`; dashboard tak punya endpoint order; frontend panel "Open Orders" **mensintesis SL/TP dari objek posisi** (palsu).
2. **Bug uang nyata di `forward.py:884-888`**: LIMIT + `GTX` (post-only) hampir selalu jadi **resting order** (tidak langsung terisi). Tapi kode baca `order.get("average") or entry` lalu **langsung daftar posisi** di `self.open` (line 1114) + pasang SL/TP (896-898) berdasar qty yang belum benar-benar terisi → posisi/SL-TP bisa yatim atau telanjang.

Pendekatan (pilihan Anda): **pending tracking + reconcile lengkap**, plus **tombol Cancel** per order.

## Perubahan

### 1. `bot/exchange.py` — wrapper open orders
Tambah method sejajar `positions()` (line 89):
```python
def open_orders(self, symbol=None) -> list[dict]:
    """Open order nyata dari Binance (limit resting + SL/TP STOP_MARKET). Dry → []."""
    if self.settings.is_dry:
        return []
    try:
        return self.client.fetch_open_orders(symbol) if symbol else self.client.fetch_open_orders()
    except Exception as e:
        log.error(f"open_orders fetch gagal: {e}")
        return []
```

### 2. `bot/forward.py` — state machine pending order (inti perbaikan)

**State baru** `self.pending: dict[str, dict]` (simbol → `{order_id, side, is_long, qty, entry, sl, tp, bet, liq, opened_ts, risk0, ...stamp}`, plus `gdecision/setup/conviction` bila gemini).

**Ubah `_live_open` (873)** — return 3-tuple `(ok, fill, pending)`:
- Buat limit order. Baca `order.get("status")`:
  - `"closed"/"filled"` langsung → pasang SL/TP, return `(True, fill, None)` (jalur lama).
  - `"open"/"new"` (resting) → **JANGAN** pasang SL/TP; return `(True, None, {order_id,...})`.
  - market order → jalur lama (langsung filled).
- Tetap jaga logika emergency-close untuk kasus filled-tapi-SL/TP-gagal.

**Ubah caller `_open_usd` (line 1095-1108)**:
- `ok, entry, pending = self._live_open(...)`.
- Bila `pending`: simpan ke `self.pending[sym]` (bukan `self.open`), catat journal `forward_open_pending`, notify "⏳ LIMIT resting". **Return** (jangan daftar self.open).
- Bila filled: jalur normal (self.open).

**Reconcile pending tiap siklus** (di `_live_reconcile`, line 926):
- `oo = self.ex.open_orders()` → map by order_id.
- Untuk tiap pending:
  - Terisi (status filled di `oo` atau posisi muncul di `positions()`) → ambil fill price, **pasang SL/TP sekarang**, pindah ke `self.open`, hapus pending, notify "✅ FILLED".
  - Cancel/expire (tak ada di `oo` & tak ada posisi) → hapus pending, notify "❌ limit batal/expired".
  - Masih resting → biarkan.
- **Timeout**: pending terlalu lama (default `pending_timeout_s`, mis. 300s, configurable via config `execution.pending_timeout_s`) → cancel order + hapus pending (jangan fallback market otomatis — skip aman). Configurable; 0 = tanpa timeout.

**Persist & restore**: sertakan `"pending"` di `_persist_state` (line 798); restore di `_restore_state` agar tahan restart bot.

**`_live_close` (917)**: juga `cancel_all_orders` + bersihkan `self.pending[sym]`.

**Status payload (`_write_status`, line 1702)**: tambah `"pending_orders"` = list ringkas `{symbol, side, type:"LIMIT", price, qty, order_id, opened_ts}`.

## Risiko & mitigasi
- **State machine = kompleksitas baru** → mitigasi: tes menyeluruh + log tiap transisi (pending→filled/canceled), notify Telegram setiap perubahan state agar bisa diaudit di live.
